## 引言：什么是 Git 和 GitHub？

- **Git** 是一种分布式版本控制系统，能追踪代码的变化，便于多人协作开发。
- **GitHub** 是基于 Git 的代码托管平台，支持协作开发、代码审查、项目管理和开源社区等功能。

---

## Git 和 GitHub 的基本概念

- **仓库（Repository）**：代码和历史记录的集合，简称“repo”。
- **分支（Branch）**：代码开发的“副本”，不同分支互不干扰，便于多人协作与新功能开发。
- **主分支（main/master）**：仓库的主线，通常保存稳定、可发布的代码。
- **Fork（派生）**：复制他人仓库到自己账户，便于独立开发与贡献代码。
- **Clone（克隆）**：把远程仓库完整下载到本地，包括代码和历史记录。
- **Commit（提交）**：保存代码的快照，便于追踪和回滚。
- **Pull Request（PR，拉取请求）**：请求将你的修改合并到主项目，是开源协作的核心机制。
- **Issue（议题）**：用于报告 Bug、建议新功能或讨论项目相关问题。

## Git 常用命令与工作流程

### 1. 克隆仓库
```bash
git clone <仓库地址>
```
> 下载远程仓库到本地

### 2. 创建分支并切换
```bash
git checkout -b <新分支名>
```
> 推荐每个新功能/修复都用独立分支，避免影响主分支

### 3. 查看状态和修改
```bash
git status
git diff
```

### 4. 添加更改并提交
```bash
git add .
git commit -m "简要说明你的改动"
```

### 5. 推送到远程仓库
```bash
git push origin <分支名>
```

### 6. 同步远程主分支最新代码
```bash
git pull origin main
```

## GitHub 协作与开源流程

1. **Fork 项目**：在 GitHub 网页点击 Fork，复制目标仓库到自己账号下。
2. **Clone 到本地**：从 Fork 后的仓库克隆代码。
3. **新建分支开发**：每个功能/修复建议使用独立分支。
4. **提交与推送**：将更改推送到自己的远程分支。
5. **发起 Pull Request**：在 GitHub 网页点击 “New Pull Request”，选择你的分支，写明改动内容。
6. **代码审查与合并**：维护者会审查代码，确认无误后合并进主项目。
7. **同步主仓库更新**：定期同步主项目代码，保持代码最新，减少冲突。

## 分支、合并与 Pull Request

- **分支（Branch）**：用于开发新功能、修复问题、做实验；主分支 main 一般保持稳定。
- **合并（Merge）**：将分支代码整合到主分支，Pull Request 成功后即完成合并。
- **Pull Request（PR）**：开源协作的基本机制，任何人都可以对公开项目发起 PR，等待维护者审核和合并。

**最佳实践：**
- 每个功能/修复单独分支，便于管理和回滚
- 在 PR 前同步主分支最新代码，减少冲突
- 提交信息要简明、准确，便于审查

## Fork 与 Clone 的区别及场景

| 功能      | Fork                      | Clone                        |
|-----------|---------------------------|------------------------------|
| 作用      | 复制仓库到自己账号        | 下载仓库到本地               |
| 是否和主仓库有关联 | 有（便于贡献）     | 没有（仅下载）               |
| 场景      | 参与开源项目、PR贡献      | 自己开发、学习、备份         |

**为什么要 Fork？**
- 便于在没权限的情况下参与和贡献开源项目
- 修改后可以发起 Pull Request 给原作者

## 常见疑问解答与最佳实践

### Q1. 直接下载源码压缩包和用 git clone 有何区别？
- 压缩包仅有一份快照，无法使用 git 管理历史、分支、合并等功能。
- git clone 可完整获得仓库历史和所有分支，是开发协作推荐方式。

### Q2. 为什么要创建分支开发？
- 避免影响主分支稳定性
- 多人协作时不同分支互不干扰
- 便于回滚和管理不同功能

### Q3. Fork 后所有改动都会合并到主项目吗？
- 不会。只有你通过 Pull Request 发起的分支内容会被维护者审核、决定是否合并。不想合并的内容不会影响主项目。

### Q4. Pull Request 可以撤回/关闭吗？
- 可以。PR 合并前可随时关闭（Close），合并后只能由维护者回滚。

### Q5. Pull Request 和 Issue 有什么区别？
- PR 是请求合并代码，Issue 是用来讨论问题、建议或任务的。

### Q6. 修改后如何只提交部分内容？
- 用分支管理不同功能，每次 PR 只选择相关分支和文件，避免无关内容被合并。

### Q7. 合并时什么是“冲突”？
- 多人修改了同一部分代码，Git 无法自动决定用哪个，需要人工处理。
